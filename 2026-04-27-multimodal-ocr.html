Multimodal OCR: Parse Anything from Documents

3B 参数统一文本与视觉符号解析，图表/化学式/UI 直接输出 SVG，开源文档解析 SOTA

问题与动机 模型架构 数据与训练 关键实验 个人思考

核心问题：视觉元素是文档中被丢弃的"另一半"

传统 OCR 管线只关注文本提取，文档中大量的图表、流程图、化学结构、UI 布局等视觉元素要么被裁成栅格图片，要么直接忽略。这些视觉符号承载了大量结构化信息，却无法被下游系统（RAG、知识库）复用。

核心思路：把文档中的视觉符号提升为与文本同等的"一等公民"，统一用一个自回归模型输出结构化表示——文本输出转录文字，可紧凑描述的图形直接输出 SVG 代码。

模型架构：dots.mocr（3B）

输出格式

对每个文档元素，模型输出一个有序元组序列 (Bk, ck, pk)：

元素类型Payload 输出示例
文本区域文字转录段落、标题、表格文本
可结构化图形SVG 代码图表、化学式、UI、流程图
复杂自然图像栅格保留照片、复杂插图

两个 checkpoint 变体

dots.mocr — 通用文档解析（文本 + 表格 + 公式 + 图形）
dots.mocr-svg — 强化 SVG 生成能力，优先把图形转为可编辑向量代码

OCR Arena：论文提出的新评测框架

用 Gemini 3 Flash 做 LLM-as-Judge，对称双向比较消除位置偏差，Elo 评分 + 1000 次 bootstrap 重采样。解决传统 OCR 规则指标（如编辑距离）的脆弱性问题——传统指标对格式差异过度敏感但对语义错误不敏感。

个人思考

对 RAG 管线的意义
传统 RAG 管线在文档解析阶段丢掉了图表信息（裁成图片后检索效果极差），这是公认的瓶颈之一。MOCR 把图表转成 SVG 代码，意味着这些视觉元素可以被当作文本来索引和检索，直接补上了 RAG 管线中"文档解析"环节最大的短板。结合目录中 OHRbench 对"文档解析质量影响 RAG 性能"的评测，MOCR 方向看起来是正解。
3B 的工程意义
模型只有 3B，但在文档解析上击败了所有开源模型、在 SVG 生成上甚至超过 Gemini 3 Pro。这说明在垂直任务上，精心设计的数据引擎 + 训练策略比堆参数更重要。对于需要本地部署 OCR 的场景，3B 意味着单卡即可推理，成本优势明显。
局限与开放问题
当前 dots.mocr 不能一次性同时输出全页文档文本 + SVG（需要分任务调用），这在实际 RAG 管线中意味着要跑两遍。另外，3B 规模下复杂表格和数学公式仍然吃力——这恰好是 T²-RAGBench 论文指出的金融文档 73% 失败来源。
与 MinerU / DISCO 的关系
目录中的 MinerU 和 DISCO 也是文档解析方向的工作。MinerU 是传统管线（规则 + 模型混合），DISCO 侧重评测。MOCR 代表的是纯端到端 VLM 路线。两条路线的融合（VLM 做主体 + 规则做兜底）可能是近期工程最佳实践。
