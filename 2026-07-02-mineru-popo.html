MinerU-Popo: Universal Post-Processing Model for Structured Document Parsing

不是再做一个 OCR，而是在 OCR 之后恢复文档级逻辑结构：跨页文本、跨页表格、标题层级、图表归属。

问题定位 四个任务 工程流程 实验与指标 个人思考

页级 OCR 和文档级结构之间还隔着一步

MinerU、PaddleOCR、GLM-OCR、Dolphin、MonkeyOCR 这类 VLM-based OCR 已经能把单页 PDF 解析成 typed blocks：标题、正文、表格、图片、caption、bbox、阅读顺序等。但下游 RAG/QA 需要的不是孤立页面，而是一个连贯的文档结构。

这篇论文的核心判断是：很多错误不是单页 OCR 识别不出来，而是 分页和复杂布局打断了文档级关系。例如上一页末尾和下一页开头其实是一段话，两个页面上的表格其实是一张跨页表，标题已经识别出来但层级不对，图片和 caption 没有关联到正确章节。

关键边界：MinerU-Popo 不是 OCR 纠错模型。它默认前序解析质量较高，不负责修复字识别错误、漏检标题、严重错列的表格 HTML 或整体阅读顺序混乱。它修的是 page-level blocks 到 document-level structure 的那一层。

一个模型，四条 task-specific pipeline

论文把后处理拆成四个子任务。模型参数可以共享，论文里是 fine-tuned Qwen3-VL-4B，但每个任务的输入过滤、prompt 和输出 schema 都不同。完整跑一次后处理不是一次 forward 同时输出所有结构，而是执行四类推理，再用代码整合。

任务模型输入模型输出用途
标题层级重识别 只保留 title / TOC-title / section-title，附带 page、bbox、content 和页面图像。 <|id|>...<|level|>...，非标题可标为 -1。 构建文档树骨架。
文本截断合并 从 text / list_item 中筛候选相邻块；内容只保留首句和尾句。 src_id -> tgt_id 的连续关系。 合并被分页、分栏、图表插入打断的段落。
跨页表格合并 相邻页边界附近的 table pair，输入表格 HTML 中的边界行和 caption 信息。 是否合并，以及 cell_list：哪些列的边界 cell 需要语义合并。 把跨页表格 HTML 拼成一个逻辑表。
图表与文本结构关联 只保留 title、image/table/chart、caption、footnote 等视觉相关块。 src_id -> tgt_id 关联边。 把 caption 挂到图表，把图表挂到所属章节。

Caption 不是图片描述

这里的 caption 指文档原文中已经存在的图注/表注，例如 Figure 2: Overview of MinerU-Popo 或 Table 3: RAG Performance。它不是模型看图后生成的图片文本化描述。代码会把 fig-caption、fig-title 归一成 image_caption，把 tab-caption、tab-title 归一成 table_caption。

过滤、分块、同步、组树

01
Task Filtering
用代码规则把 OCR blocks 筛成每个任务的候选输入，减少噪声和视觉/文本上下文长度。
02
Dynamic Chunking
长文档按页切 chunk，在边界附近选择目标元素密集的页，避免一次塞完整 PDF。
03
Synchronization
标题层级用重叠区域计算偏移，校准后续 chunk 的 level；其他任务主要取并集。
04
Assembly
代码把四类关系写回 blocks，再组装成树，并进一步做 subnode 和摘要。

文本合并规则

源码里的文本候选不是只限跨页，同页中被图片、表格、分栏隔开的两段也可能被筛到。它会拒绝一些明显或疑似不该合并的情况：前段以句号/问号/冒号/分号等终止，前段太短，后段像列表项，任一段包含 tab，或两段都以数字开头。这里的 tab 是 \t 制表符，不是普通空格。

表格合并规则

表格候选更依赖上游 HTML。代码只看相邻页的“上一页最后一个 table”和“下一页第一个 table”，再检查中间是否夹正文、caption 是否一致、下表 caption 是否有 continued/续表 标记、footnote 数量、bbox 宽度差是否小于 10%、列结构是否兼容。列结构来自 table HTML，使用 BeautifulSoup 解析 tr、td、th、rowspan、colspan。

代码和论文的小差异：论文写的是 three-page overlap；开源 adaptive_chunk 默认是 chunk_size=50、overlap=1，并在目标页前后 5 页内选择任务元素最多的页作为边界。

最终组树是代码逻辑

模型输出的是局部关系：level、contd、image、table_merge、cell_list。最终 get_json_tree.py 通过确定性逻辑合并跨页表、聚合文本块、用标题 level 栈建树、把图表和 caption/footnote 加回正确章节。语义增强中，长文本节点超过约 500 字符会被切成 subnode；摘要生成则由 LLM 另一步完成。

TEDS 是什么

TEDS 是 Tree Edit Distance Similarity，用树编辑距离衡量两棵结构树有多像。标题层级天然可以表示成树；表格也可以通过 HTML/DOM 表示成树，例如 table -> tr -> td，并把 rowspan、colspan 和单元格文本作为结构/内容的一部分比较。

开源 eval/evaluate.py 主要评标题层级 TEDS：先把预测标题块和 GT 标题块按 page、bbox、文本相似度和类型对齐，再用 zss 树编辑距离库计算 content-aware TEDS。论文里的文本合并、图文关联、表格合并和 RAG/QA 评测脚本没有完整开源。

现象解释
结构树 RAG 多数子集更准且更快用标题路径 + node summary 做检索，减少全文 embedding 和无关上下文。
工业/金融细节题可能不占优summary 更适合关键观点、方法、趋势，不适合精确数字和复杂表格细节。
Visual RAG 召回高但答案未必好页级图像检索命中后 bbox recall 高，但高分辨率密集页面会压迫 VLM 理解。

个人思考：工程可控，但边界很清楚

Thought 01

标题漏检是硬边界。代码只把已经被上游标成 title/TOC-title/section-title 的块送进标题层级任务。如果 OCR 一开始把标题识别成正文，Popo 默认不会把它重新发现出来。它能修“层级错”，不能稳定修“标题漏”。

Thought 02

规则过滤偏保守。规则可以降低成本、减少误合并，但源码里有些规则并非完全确定，例如“两个文本都以数字开头就拒绝”“下表 caption 没有 continued 就拒绝”。这些规则可能系统性降低召回，更适合作为模型输入特征，而不是 hard filter。

Thought 03

表格任务高度依赖上游 HTML。列数、rowspan/colspan 和边界 cell 合并都来自表格 HTML。如果上游表格 HTML 已经严重错列、漏行或解析失败，Popo 不会重新从图像中恢复真实表格结构。

Thought 04

这篇的价值在“文档级关系恢复”。它不是端到端最强文档理解，而是把 OCR 后处理拆成几个可控的结构关系预测，再由代码组树。这种设计适合生产落地，也暴露了一个改进方向：保留确定规则，放松弱规则，用模型做更高召回的二阶段判断。

一句话总结：MinerU-Popo 默认 OCR 内容已经可用，它补的是文档结构这层：把 page-level blocks 变成 document-level tree。它的亮点是工程化拆解，风险也来自工程化拆解中过早、过硬的规则筛选。
