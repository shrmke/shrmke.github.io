From BM25 to Corrective RAG

2026-04-03 · arXiv: 2604.00491 · Text + Table Retrieval

T²-RAGBench
Retrieval Strategy

测试集速记

  • Benchmark：T²-RAGBench（23,088 个查询，7,318 份金融文档，SEC 10-K/10-Q + 年报表格）。
  • 数据形态：文本与精确数字表格混合，文档切成 text chunk + markdown 表格单元。
  • 评测指标：Recall@{5,20}、MRR@{3,10} 以及 Number Match，覆盖检索与生成链路。

论文比较了 10 条检索路线，涵盖 BM25、text-embedding-3-large、混合 RRF、HyDE、Multi-Query、Context Retrieval、CRAG（Corrective Retrieval Augmented Generation）等。

作者结论

  • 两阶段管道最佳：BM25 + 稠密检索 RRF 融合，再用 Cross-Encoder 重排序 → Recall@5 = 0.816，MRR@3 = 0.605，显著领先所有单阶段方法。
  • BM25 仍是基石：在金融语料上 BM25 几乎全面压过 text-embedding-3-large，体现精确术语/实体匹配的重要性。
  • 查询扩展收益有限：HyDE 和 Multi-Query 基本无增益，HyDE 甚至拖低 Recall@5（0.544）。
  • Context Retrieval 值得保留：索引时前置 LLM 摘要带来 2~3pp 的稳定提升。
  • CRAG 只带来小幅收益：自适应检索 Recall@5 = 0.658，略好于 BM25，但仍落后于简单的混合检索（0.695）。
  • 表格仍是主要瓶颈：73% 的失败来自表格结构不匹配；作者建议结合结构化解析或专门的表格检索器。

实务建议：先用混合检索 + Cross-Encoder 重排序，再考虑 Context Retrieval；对需要精确数字的场景谨慎使用 HyDE，且务必在领域数据上实测，不要仅依赖通用榜单（MTEB/BEIR）的排名。
