Beyond Semantic Similarity: Rethinking Retrieval for Agentic Search via Direct Corpus Interaction

不是再训练一个更强的检索器，而是把检索接口从 top-k 相似度切片改成高分辨率的终端交互。

核心观点 怎么搜索 上下文管理 实验结果 个人思考

把检索看成 Agent 与语料库之间的“接口设计”

传统 RAG 把文档切块、向量化，然后让检索器返回 top-k。这个接口很高效，但它把语料库压缩成一次相似度排序：Agent 只能看到检索器愿意暴露的片段。如果第一步没有召回关键证据，后面的推理再强也很难补救。

本文提出 Direct Corpus Interaction, DCI：Agent 不调用 embedding 模型、向量索引或检索 API，而是直接在原始语料库上使用终端工具搜索，比如 rg、grep、find、sed、head、tail 和轻量脚本。

Retrieval Interface Resolution

论文的关键概念是 检索接口分辨率：接口能否让 Agent 在比“整篇文档”或“固定段落切片”更小、更精确的单位上操作。DCI 的优势不是一定找到更多 gold documents，而是当它摸到一个有用文档后，可以继续做行级、正则级、上下文级定位。

接口Agent 看到什么适合什么主要风险
Retriever-mediated 按相似度排序的 top-k 文档或片段 大规模、静态、需要低延迟的语料 早期过滤不可逆；复杂精确约束容易被向量相似度抹平
Direct Corpus Interaction 命令行工具输出：路径、匹配行、上下文、计数、局部文本 中等规模、本地化、动态变化、需要深度调查的语料 搜索宽度扩大时成本和延迟迅速上升
一句话：这篇论文不是说“向量检索没用”，而是说对于能规划、试错、验证的强 Agent，固定 top-k 检索接口可能太低分辨率；grep/bash 这种原始接口反而给了 Agent 更大的行动空间。

DCI 不是固定流程，而是边搜边改

它没有规定“一定先搜文件名，再搜文件内容”。成功轨迹更像人类在代码仓库或本地资料里查问题：先用宽关键词找锚点，再把多个条件串起来过滤，最后围绕候选文件做局部验证。

01
发现锚点
用核心实体、年份、术语直接全库搜索，先找到可能相关的文件和行号。
02
迭代缩小
把多个弱线索组合成强约束，例如 A 且 B，或者在某个路径下搜索。
03
局部阅读
只读匹配行附近的上下文，而不是整篇打开；必要时进入单文件内深搜。
04
交叉验证
在多个候选文档之间来回确认约束，排除同名实体、时间错位和伪线索。

典型命令形态

rg -n "keyword" @corpus/ | head rg -n "keyword1" @corpus/ | rg "keyword2" rg -n -C 5 "exact phrase" candidate.txt find @corpus/ -type f | rg "2024|report|team"

附录中的 DCI-Agent-Lite 轨迹统计很能说明问题：在 100 个案例、3,168 条命令里，rg | head 占 56.2%，rg | rg 占 20.6%，wc 占 7.8%，单关键词 rg 占 6.3%，ls 占 4.3%，而 cat 只占 0.1%。这说明 Agent 主要把 shell 当作“高分辨率搜索界面”，而不是把整篇文档搬进上下文。

关于“10 万文档”：BrowseComp-Plus 语料是 100,195 个文档，平均长度 5,179 个 whitespace-split words。这里的规模主要是文档/文件数量，不是 PDF 页数。单个长文档可以靠行号和局部读取处理；真正伤 DCI 的是候选文件数量太多带来的搜索宽度。

长程搜索必须控制工具输出

DCI 会产生大量 grep、rg、文件读取结果。如果全部保留，几十轮后上下文会被工具输出撑满。论文给 DCI-Agent-Lite 设计了三类运行时上下文管理机制。

机制发生时机做什么代价
Truncation 单次工具返回时 把工具输出按字符数截断；L1 是 50K chars，L2-L4 是 20K chars 可能截掉后部证据，但能防止单次输出占满窗口
Compaction 历史工具输出累计过多时 超过 240K chars 后，清空较早工具结果，只保留短占位符和最近 12 轮 零 LLM 成本，但会丢掉历史原文
Summarization 压实后仍有上下文压力时 用模型把被压实历史变成摘要，并保留最近 20K tokens 额外推理成本；摘要可能丢掉路径、行号、关键词等关键细节

规模压力测试

作者把 BrowseComp-Plus 从 100K 文档扩到 200K 和 400K，方法是注入 FineWeb distractors。100K 到 200K 时，DCI-Agent-CC 平均工具调用从 38.5 增到 86.9，延迟和成本都超过翻倍，准确率下降 13.6 点。到 400K 文档时，准确率掉到 37.5%，平均工具调用到 122.4，并且 20 个样例达到最大工具预算后终止。

实验边界：论文成本主要是 agent-side API cost，不包括本地执行 grep、文件系统扫描、CPU/IO 的基础设施成本。因此“更便宜”不等于“端到端系统一定更便宜”，尤其在大规模语料和高并发场景。

个人判断：更像 RAG 系统的第二层接口

Takeaway 1

这篇论文最有价值的地方不是“grep 打败 embedding”这个口号，而是把问题从“选哪个检索模型”推进到“给 Agent 什么样的语料库操作界面”。对于复杂任务，接口分辨率直接决定 Agent 能不能验证局部证据、组合稀疏线索、修正假设。

Takeaway 2

我不认为 DCI 会直接替代向量库。更现实的产品形态是混合式：先用 BM25/向量检索/元数据筛选找到候选区域，再把候选文件树、全文搜索、上下文读取、正则过滤暴露给 Agent，让它在小范围内做 DCI 式深挖。

Takeaway 3

对于企业内部文档、代码仓库、审计资料、PDF 转文本后的本地 corpus，DCI 很有启发。与其只做 chunk embedding，不如同时保留原始文件、路径、标题层级、行号、页码和全文搜索接口。Agent 要的是可操作的工作区，而不只是相似段落。

Caveat

这仍然是研究型方案。BrowseComp-Plus 的单 query 延迟可以非常高，不适合实时问答；大规模扩展也明显吃力。另外，论文使用了若干 2026 年的闭源模型和 LLM judge，复现性依赖这些模型与评测设置。
